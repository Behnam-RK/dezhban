Two ways to cut the wire

dezhban has one primary mode and one fallback. The VPN guard is always on — it enforces the entire time, so if the tunnel drops, nothing leaks. The country-blocklist is a best-effort fallback for hosts with no VPN: it only reacts after the next poll, so it can shrink the leak window but never close it. The difference lives in the resting state — the guard is armed even when all is well; the fallback is simply off.

The guard, and the fallback

Primary interface-aware · always-on

VPN / full-tunnel guard

Watches your tunnel. Blocks by network interface. The GUARD baseline is enforced the whole time, so a tunnel drop is cut instantly — followed, by default, by a bounded automatic reconnect window (vpn.reconnectWindow, 30 s) so the VPN can redial any server with zero interaction; set it to "0" for a strict zero leak window. Use it whenever you're behind a full-tunnel VPN. Details: modes.md § Automatic reconnect window.
GUARD ALWAYS ON — only the tunnel may egress resting · armed
▼VPN exit country forbidden · or country unknown
FULL-BLOCK tunnel egress cut — endpoint kept open to reconnect
▲windowed probe: exit allowed again → back to GUARD
GUARD ruleset 
# dezhban ruleset — default-deny outbound.
pass quick on lo0 all no state
pass out quick on { utun4 } all no state
pass out quick to { 178.252.141.142 } no state
block drop out all
FULL-BLOCK ruleset 
# dezhban ruleset — default-deny outbound.
pass quick on lo0 all no state
pass out quick to { 178.252.141.142 } no state # reconnect path
block drop out all

Leak window: 0 s. The guard is standing policy, not a poll result — there is no moment where physical egress is permitted, so a tunnel drop cannot leak.

Fallback destination-aware · reactive

Country-blocklist

Watches your public IP's country. Blocks by destination. For hosts not behind a full tunnel — while your country is allowed it applies no rules at all.
ALLOW firewall OFF — network fully open resting
▼public IP country ∈ blocklist ×hysteresis · or unknown & failClosed
BLOCK default-deny + DNS/geo allowlist
▲country leaves blocklist ×hysteresis → back to ALLOW
ALLOW ruleset 
∅  no rules installed — outbound is wide open
BLOCK ruleset 
# dezhban ruleset — default-deny outbound.
pass quick on lo0 all no state
pass out quick proto { udp tcp } to { 1.1.1.1 8.8.8.8 } port 53
pass out quick to { <geo-API provider IPs> } no state
block drop out all

Lowering pollInterval (even to 3 s) shrinks the leak window — it never closes it. A transition commits only after hysteresis × pollInterval, and while the country looks fine there are no rules at all.

Which mode do I want?

If your VPN silently drops, do you care that traffic keeps flowing on your real connection — exposing your real IP?

▸ Yes, I care

→ VPN guard recommended

GUARD is always on, so a drop is cut in 0 seconds. True regardless of whether your provider ever routes you through a forbidden country — that only decides whether FULL-BLOCK ever fires; otherwise you just stay in GUARD.

▸ No, I don't

→ Country-blocklist

The fallback. Only meaningful if the country you block is your real physical location — and it can't promise zero leak at any poll interval.

Case A — physically inside the forbidden country

On a drop, your real IP resolves to the blocked country, so the fallback eventually blocks — but only after the next poll, and the detection poll itself travels over the exposed line.

≤ 90 sleak · pollInterval × hysteresis

Case B — physically in an allowed country

On a drop, your real IP resolves to an allowed country, so the fallback stays in ALLOW and never blocks. Your real IP leaks indefinitely.

∞leak · the fallback is blind here

Being sure your provider has no servers in the forbidden country only cancels FULL-BLOCK — you simply stay in GUARD the whole time. You lose nothing by choosing the guard; keep blockedCountries listed anyway as a free backstop. Don't run guard mode without a VPN, though — GUARD needs a tunnel to pass traffic, which is why it's a deliberate opt-in.
